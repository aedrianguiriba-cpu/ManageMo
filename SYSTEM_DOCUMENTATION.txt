OVERVIEW
--------
ManageMo is a web-based inventory and asset management system built for
Pampanga State University (PSU). It runs on PHP (no framework, no database)
using hardcoded/mock data arrays. The UI uses Bootstrap 5.3 and Font Awesome 6.4.
All pages are server-rendered PHP. Session-based authentication (BCrypt passwords).

No external database is required — all data lives in:
  config/data.php       — live data functions (getUsers, getInventory, etc.)
  config/mock_data.php  — MockData class with the same data (legacy/reference)
  config/functions.php  — all business logic helpers
  config/constants.php  — app-wide constants (roles, statuses, urgency levels)

MODULE DOCUMENTATION

────────────────────────────────────────────────────────────────────────────────
MODULE 1: AUTHENTICATION  (index.php, auth.php, logout.php)
────────────────────────────────────────────────────────────────────────────────
Flow:
  1. User visits index.php — shown login form (email + password).
  2. Form POSTs to auth.php.
  3. auth.php calls getUsers(), finds user by email, verifies BCrypt hash via
     password_verify(). On success: sets $_SESSION['user_id'], redirects to
     role-appropriate dashboard (/admin/dashboard.php or /user/dashboard.php).
  4. logout.php calls session_destroy(), redirects to index.php.

Role system:
  ROLE_ADMIN = 'admin'  → admin/ pages only
  ROLE_USER  = 'user'   → user/ pages only
  requireAdmin() / requireUser() guard every page and redirect if role mismatch.

Security:
  - Passwords hashed with BCrypt (PASSWORD_BCRYPT).
  - All user-supplied values passed through sanitizeInput() (htmlspecialchars + trim).
  - File uploads validated by MIME type (finfo) + extension + size ≤ 5 MB.

────────────────────────────────────────────────────────────────────────────────
MODULE 2: INVENTORY  (user/inventory.php, admin/inventory.php, admin/inventory-campus.php)
────────────────────────────────────────────────────────────────────────────────
Data source: getInventory() → array of items.
Each item has:
  id, qr_code_id, item_name, category, description,
  campus_id, quantity, status, location,
  purchase_date, cost, condition, created_at

Item statuses: available | borrowed | maintenance | disposed

QR Code system:
  - Every item has a base qr_code_id (e.g. 'QR-A1B2C3D4E5').
  - getItemUnitQRCodes($item) generates per-unit IDs:
      quantity=3 → ['QR-A1B2C3D4E5-U01', 'QR-A1B2C3D4E5-U02', 'QR-A1B2C3D4E5-U03']
  - lib/qrcode.php generates the actual QR image for each code.
  - QR images are stored in assets/qrcodes/.

User view: read-only browse, filter by campus/category.
Admin view: full CRUD — add, edit, delete items; update status; generate QR codes.
Campus view: admin/inventory-campus.php filters items by campus_id.

────────────────────────────────────────────────────────────────────────────────
MODULE 3: REQUEST SUBMISSION  (user/requests.php)
────────────────────────────────────────────────────────────────────────────────
Three request types (selected via radio buttons):
  a) Borrow     — borrow an existing inventory item temporarily
  b) Item       — request procurement of a new item
  c) Service    — request maintenance/repair/service on an existing item

Cart system (client-side JS):
  - addToCart(id, name)     — adds item to cart array
  - removeFromCart(index)   — removes item by index
  - renderCart()            — renders cart list in sidebar
  - resetStaging()          — clears cart when request type changes
  - On submit: serializes cart to hidden input items_json as JSON

Form fields (vary by type):
  Borrow:  inventory item, expected return date, urgency, reason
  Item:    item description, quantity, urgency, reason
  Service: inventory item, service type, description, urgency

Required for all types:
  Approval letter upload (PDF/JPG/PNG, max 5 MB) from office/college/dept head.

Server-side:
  - MIME + extension + size validation on upload.
  - File saved to assets/uploads/approval_letters/ with timestamped random name.
  - Request stored (mock: session flash message shown on redirect).

────────────────────────────────────────────────────────────────────────────────
MODULE 4: REQUEST MANAGEMENT / APPROVAL  (admin/requests.php)
────────────────────────────────────────────────────────────────────────────────
Data source: getRequests() → array of all requests.
Each request has:
  id, request_number, user_id, request_type (borrow/item/service),
  inventory_id (nullable for item requests), service_description,
  urgency (low/medium/high/critical), status, reason_for_request,
  approval_notes, approved_by, approved_at, expected_return_date,
  created_at, updated_at, full_name, email, item_name, qr_code_id

Request lifecycle:
  pending → approved / disapproved → (for approved item requests) → delivered

Admin actions:
  - Filter by type, status, urgency, campus.
  - Approve with optional notes → status = 'approved'.
  - Disapprove with reason → status = 'disapproved'.
  - Mark as delivered (triggers delivery email via sendDeliveryEmail()).
  - View attached approval letter.

Email notifications:
  sendDeliveryEmail($email, $name, $request_number, $stage)
    stages: 'out_for_delivery' | 'pickup_ready'
  Uses PHP mail() — requires a working SMTP/sendmail config.

────────────────────────────────────────────────────────────────────────────────
MODULE 5: MY RECORDS  (user/borrow-records.php)
────────────────────────────────────────────────────────────────────────────────
Three tabs via ?tab=borrow|item|service (default: borrow).
Status filter per tab via ?status= (hidden input preserves tab on form submit).

Tab 1 — Borrow Records:
  Source: getBorrowRecords() filtered by user_id.
  Each record merged with inventory item (item_name, qr_code_id).
  Columns: Item, QR Code, Borrowed, Expected Return, Returned On, Status, Notes.
  Status badges: active (blue) | returned (green) | overdue (red).
  Overdue detection: active record where expected_return_date < now().
  Stats: Active / Returned / Overdue / Total.
  Overdue alert banner shown if stat_borrow_overdue > 0.

Tab 2 — Item Requests:
  Source: getRequests() filtered by user_id + request_type='item'.
  Columns: Request #, Item/Description, Urgency, Reason, Submitted, Status, Admin Notes.
  Status badges: pending (amber) | approved (green) | disapproved (red).
  Stats: Pending / Approved / Disapproved / Total.

Tab 3 — Service Requests:
  Source: getRequests() filtered by user_id + request_type='service'.
  Columns: Request #, Item, Service Type, Description, Urgency, Submitted, Status.
  Stats: Pending / Approved / Disapproved / Total.

Print: window.print() — @media print hides sidebar/topbar/tabs/filters, renders
       table cleanly with black text on white background.

MODULE 6: QR DELIVERY SCANNER  [PLANNED — MOBILE APP FEATURE]
────────────────────────────────────────────────────────────────────────────────
NOTE: QR code scanning is NOT implemented in this web application.
      It is reserved for a dedicated companion mobile app to be developed
      separately (e.g. React Native, Flutter, or native Android/iOS).

Purpose: Allow the recipient user to confirm delivery of approved item/borrow
requests by scanning the QR code printed on each delivered unit. When all
expected units are scanned, the request is automatically marked as 'delivered'.

Planned Mobile App Flow:
  1. User opens the ManageMo mobile app and logs in with the same credentials.
  2. App fetches the user's approved requests from the ManageMo API.
  3. User selects the request they are receiving.
  4. App activates the device rear camera to scan QR codes.
  5. Each scanned code is matched against the expected per-unit QR codes for
     that request (e.g. QR-A1B2C3D4E5-U01, QR-A1B2C3D4E5-U02 ...).
  6. Matched items are checked off; a progress indicator shows X of Y scanned.
  7. When all units are confirmed → app calls the ManageMo API endpoint
     (POST /api/requests/{id}/deliver) → request status updated to 'delivered'.
  8. Duplicate scans are rejected silently; unrecognised codes show a warning.

QR Code Format (already defined in the web system):
  - Every inventory item has a base qr_code_id, e.g. 'QR-A1B2C3D4E5'.
  - getItemUnitQRCodes($item) generates per-unit codes:
      quantity=3 → ['QR-A1B2C3D4E5-U01', 'QR-A1B2C3D4E5-U02', 'QR-A1B2C3D4E5-U03']
  - QR images are generated by lib/qrcode.php and stored in assets/qrcodes/.
  - For item requests with no inventory_id, the request_number itself is used
    as the QR payload printed on the delivery paperwork.

Planned Technology Stack (Mobile App):
  - Framework:  React Native, Flutter, or native Android/iOS
  - QR Library: react-native-camera / mobile_scanner (Flutter) / ZXing (Android)
  - Auth:       Same JWT or session token as the web system
  - API:        REST API layer to be added to ManageMo backend

MODULE 7: BORROW MANAGEMENT  (implicit via borrow-records + admin/requests)
────────────────────────────────────────────────────────────────────────────────
Data source: getBorrowRecords() → array of borrow tracking records.
Each record has:
  id, inventory_id, user_id, request_id,
  borrow_date, expected_return_date, actual_return_date (null until returned),
  status (active/returned/overdue), notes, created_at

Overdue logic: status='active' AND expected_return_date < today.
Return process: admin sets actual_return_date → status = 'returned' →
                inventory item status reverts to 'available'.

TECH STACK
================================================================================

  Backend:    PHP 8.x (no framework, no ORM)
  Data:       Hardcoded PHP arrays (no database)
  Auth:       PHP sessions + BCrypt (password_hash / password_verify)
  Frontend:   Bootstrap 5.3, Font Awesome 6.4
  QR Reader:  [MOBILE APP — planned, not in web]  jsQR / ZXing / mobile_scanner

================================================================================
KNOWN LIMITATIONS (MOCK MODE)
================================================================================

  - All data mutations (approve, submit, deliver, etc.) are NOT persisted
    between page loads. Session flash messages confirm the action visually,
    but the underlying PHP arrays are reset on each request.
  - Email delivery requires a working PHP mail() / SMTP configuration.
  - QR scanner requires HTTPS or localhost for camera access (browser security).
  - No real file storage across requests for uploaded approval letters.
